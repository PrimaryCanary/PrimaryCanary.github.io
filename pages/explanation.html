Remembering Input

The extension has been chosen; DFAs will gain an integrated stack to help them learn to count. Now let‚Äôs dive into the specifics of the integration. States will remain unchanged; they already represent individual bits of memory so there is no need to extend them with more. Transitions will not remain unchanged; just like how transitions were the mechanism by which DFAs made decisions, they will now additionally be the mechanism by which PDAs decide how to manipulate stacks. Transitions will now have three parts:

  1. An input to consume. This part remains unchanged from DFAs.
  2. Zero or more items to push to the stack.
  3. Zero or more items to pop from the stack.

We‚Äôll write them like this: --input, push operations, pop operations-->. Multple stack operations are written by concatenating them together e.g.¬†pushing three separate zeroes and two ones is written --input, 000, 11-->. It‚Äôs common for the push or pop operations to be empty, and leaving them blank is kind of ugly. To fix it, empty operations are written as Œµ.

These stack operations are all it takes for the new automata to be able to count.

That‚Äôs all it takes?

Definitely. Consider the previous example of programming language brackets. PDAs can count how many opening brackets ‚Äú{‚Äù occur by pushing one to the stack each time one is encountered and popping one for each ‚Äú}‚Äù.

Making Decisions

PDAs can now remember input but they still can‚Äôt use that information. We need to allow PDAs to make decisions based on the status of the stack in order for the newfound memory to be meaningful. We let PDAs accept or reject an input based on two new criteria:

  1. Popping from an empty stack is an instant reject. This is similar to the trash rule in a DFA.
  2. The stack must be empty for an input to be accepted.

These rejection criteria give them a notion of equality. So to answer your question Nibbles, if there are too few ‚Äú}‚Äù, the stack won‚Äôt be empty and the input will be rejected. If there are too many ‚Äú}‚Äù, the PDA will try to pop from an empty stack and be rejected.

Input

We‚Äôve been especially vague about what input is. The definition up until now has been ‚Äúany text at all‚Äù. No longer. Formally, the input is a string consisting of zero or more characters from a finite set Œ£\Sigma, called the input alphabet. Only elements of Œ£\Sigma may appear in the input.

So no trash rule.

Correct. There is a similar set Œì\Gamma called the stack alphabet, with one additional requirement: Z‚ààŒìZ\in\Gamma. ZZ is how we denote the an empty stack. This has the corollary that every PDA will push a ZZ somewhere early on and check for ZZ near the end. Only elements of Œì\Gamma may appear in the stack. Finally, the set of all inputs accepted by the PDA is called the language LL.

States

States are modelled as a finite set QQ. Conventionally, the start state is unambiguously labelled q0q_0 with the requirement q0‚ààQq_0\in Q. Accepting states are also modelled as a finite set FF with the requirement F‚äÜQF\subseteq Q. The collection of rejecting states doesn‚Äôt usually get an explicit name but can be inferred with Q\FQ\setminus F.
